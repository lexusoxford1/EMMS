System Workflow

1. Employee performs attendance

The employee records `Morning In`, `Morning Out`, `Afternoon In`, and `Afternoon Out` in strict sequence from the Attendance page.

2. Browser captures location

Before the request is sent, the browser captures latitude and longitude using `navigator.geolocation`.

3. Frontend sends API request

`attendance.js` submits the attendance payload to the backend through `fetch` with CSRF protection.

4. Backend validates data

Django validates authentication, JSON structure, coordinate limits, and the next valid attendance step.

5. Database stores records

The system updates the daily `Attendance` record and creates a matching `AttendanceLocation` log entry.

6. Admin panel monitors output

Admins review the resulting attendance location records from the daily Location page and the history API.

Validation and Access Rules

Authentication: Protected endpoints require authenticated access. The live web app uses secure Django session-based login, while Swagger documents that access without exposing raw session cookies.
Employee access: Only authenticated employee users can access employee attendance APIs and record their own attendance actions.
Admin access: Only authenticated admin users can access admin monitoring APIs and location monitoring records.
Coordinates: Latitude must be between -90 and 90; longitude must be between -180 and 180.
Workflow: The system only accepts the next valid step in the order Morning In, Morning Out, Afternoon In, Afternoon Out.
Payload: The request body must be valid JSON with latitude and longitude values.
History access: Employees only see their own records; admins can review broader monitoring data.
